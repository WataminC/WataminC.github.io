    Stronger Security Notions

    CPA-security

    Nowadays, this is the minimal notion of security an encryption scheme should satisfy.

    • Fix Π,A\Pi, AΠ,A
    • Define a randomized exp't PrivCPAA,Π(n)PrivCPA_{A, \Pi}(n)PrivCPAA,Π​(n):
    1. k ←Gen(1n)\leftarrow Gen(1^n)←Gen(1n)
    2. A(1n1^n1n) interacts with an encryption oracle Enck(.)Enc_k(.)Enck​(.), and then ouputs m0,m1m_0, m_1m0​,m1​ of the same length
    3. b←0,1,c←Enck(mb)b \leftarrow {0, 1}, c \leftarrow Enc_k(m_b)b←0,1,c←Enck​(mb​), give c to A
    4. A can continue to interact with Enck(.)Enc_k(.)Enck​(.)
    5. A outputs b′b'b′; A succeeds if b = b', and experiment evaluates to 1 in this case

    Π\PiΠ is secure against chosen-plaintext attacks (CPA-secure) if for all PPT attackers A, there is a negligible function ϵ\epsilonϵ such that

    Pr[PrivCPAA,Π(n)=1]≤12+ϵ(n) Pr[PrivCPA_{A, \Pi}(n) = 1] \leq \frac{1}{2} + \epsilon(n) Pr[PrivCPAA,Π​(n)=1]≤21​+ϵ(n)

    Pseudorandom functions and Block ciphers

    Pseudorandom functions(PRFs)

    • F is a pseudorandom function if FkF_kFk​, for uniform key k∈{0,1}nk \in \{0, 1\}^nk∈{0,1}n, is indistinguishable from a uniform function f∈Funcnf \in Func_nf∈Funcn​

    • Formally, for all poly-time D:

    ∣Prk←{0,1}n[DFk(.)=1]−Prf←Funcn[Df(.)=1]∣≤ϵ(n) |Pr_{k \leftarrow \{0, 1\}^n}[D^{F_k(.)} = 1] - Pr_{f \leftarrow Func_n}[D^{f(.)} = 1]| \leq \epsilon(n) ∣Prk←{0,1}n​[DFk​(.)=1]−Prf←Funcn​​[Df(.)=1]∣≤ϵ(n)

    Block ciphers

    Block ciphers are practical constructions of pseudorandom permutations

    • No asymptotics:n F:{0,1}n×{0,1}m→{0,1}mF: \{0, 1\}^n \times \{0, 1\}^m \rightarrow \{0, 1\}^mF:{0,1}n×{0,1}m→{0,1}m

      • n = "key length"
      • m = "block length"

    AES

    Advanced encryption standard (AES)

    • Block length = 128bits
    • Key length = 128, 192, 256 bits

    Modes of Encryption

    CTR Mode

    • Choose ctr←{0,1}nctr \leftarrow \{0, 1\}^nctr←{0,1}n, set c0=ctrc_0 = ctrc0​=ctr
    • For i = 1 to t:
      • ci=mi⊕Fk(ctr+i)c_i = m_i \oplus F_k(ctr + i)ci​=mi​⊕Fk​(ctr+i)
    • Output c0,c1,⋯ ,ctc_0, c_1, \cdots, c_tc0​,c1​,⋯,ct​

    Ciphertext expansion is just 1 block

    Theorem: If F is a pseudorandom function, then CTR mode is CPA-secure.

    CBC mode

    • Enck(m1,m2,⋯ ,mt)Enc_k(m_1, m_2, \cdots, m_t)Enck​(m1​,m2​,⋯,mt​)

      • Choose random c0←{0,1}nc_0 \leftarrow \{0, 1\}^nc0​←{0,1}n (also callled the IV)
      • For i = 1 to t:
        • ci=Fk(mi⊕ci−1)c_i = F_k(m_i \oplus c_{i-1})ci​=Fk​(mi​⊕ci−1​)
      • Output c0,c1,⋯ ,ctc_0, c_1, \cdots, c_tc0​,c1​,⋯,ct​

    • Decryption requires F to be invertible

    • Ciphertext expansion is just 1 block

    Theorem: If F is a pseudorandom function, then CBC mode is CPA-secure

    ECB mode

    • Enck(m1,⋯ ,mt)=Fk(m1),⋯ ,Fk(mt)Enc_k(m_1, \cdots, m_t) = F_k(m_1), \cdots, F_k(m_t)Enck​(m1​,⋯,mt​)=Fk​(m1​),⋯,Fk​(mt​)
    • Deterministic
      • Not CPA-secure!

    Security Against Chosen-Ciphertext Attacks

    Malleability

    (Informal:) A scheme is malleable if it is possible to modify a ciphertext and thereby cause a predictable change to the plaintext

    All the schemes we have seen so far are malleable!

    example: flip bit when one time pad

    CCA-security

    • Define a randomized exp't PrivCCAA,Π(n):PrivCCA_{A, \Pi}(n):PrivCCAA,Π​(n):
    1. k←Gen(1n)k \leftarrow Gen(1^n)k←Gen(1n)
    2. A(1n)A(1^n)A(1n) interacts with an encryption oracle Enck(.)Enc_k(.)Enck​(.), and a decryption oracle Deck(.)Dec_k(.)Deck​(.), and then outputs m0,m1m_0, m_1m0​,m1​ of the same length
    3. b←{0,1}b \leftarrow \{0, 1\}b←{0,1}, c←Enck(mb)c \leftarrow Enc_k(m_b)c←Enck​(mb​)
    4. A can continue to interact with Enck(.),Dec(.)Enc_k(.), Dec(.)Enck​(.),Dec(.), but may not request decryption of c
    5. A outputs b'; A succeeds if b = b', and experiment evaluates to 1 in this case.
    • Π\PiΠ is securee against chosen-ciphertext attacks(CCA-secure) if for all PPT attackers A, there is a negligible function ϵ\epsilonϵ such that

    Pr[PrivCCAA,Π(n)=1]≤12+ϵ(n) Pr[PrivCCA_{A, \Pi}(n) = 1] \leq \frac{1}{2} + \epsilon(n) Pr[PrivCCAA,Π​(n)=1]≤21​+ϵ(n)

    • If a scheme is malleable, then it cannot be CCA-secure
    • CCA-security implies non-malleability
